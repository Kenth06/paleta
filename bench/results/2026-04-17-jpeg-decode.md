# Bench results — 2026-04-17 (DC-only vs jSquash head-to-head)

**Headline: paleta DC-only is 3–9× faster than mozjpeg (via jSquash) on real
JPEGs once the 256-entry Huffman fast lookup is in place.**

## Setup

- Node v22, aarch64 (Apple Silicon), vitest 2.1.9.
- WASM artifact built with `+simd128`, wasm-opt -O3.
- paleta's Rust DC-only decoder vs `@jsquash/jpeg@1.6.0` (mozjpeg), both
  running inside the same Node process via `await init(WebAssembly.Module)`.
- Fixtures generated by Python PIL at quality 80.

## Decode time (single call)

| Fixture                       | paleta DC-only | jSquash full | Δ         |
| ----------------------------- | -------------- | ------------ | --------- |
| 64×64 solid 4:4:4             | **0.008 ms**   | 0.035 ms     | **4.56×** |
| 128×128 four-quadrants 4:2:0  | **0.014 ms**   | 0.119 ms     | **8.69×** |
| 640×480 scene 4:4:4           | **0.801 ms**   | 2.488 ms     | **3.11×** |
| 640×480 scene 4:2:0           | **0.469 ms**   | 2.179 ms     | **4.65×** |
| 1280×720 scene 4:2:0          | **1.152 ms**   | 6.281 ms     | **5.45×** |

## End-to-end `getPalette` time (decode + resize + histogram + Wu)

| Fixture              | DC-only path | full-decode path | Δ         |
| -------------------- | ------------ | ---------------- | --------- |
| 640×480 4:2:0        | **0.683 ms** | 2.408 ms         | **3.53×** |
| 1280×720 4:2:0       | **1.412 ms** | 6.710 ms         | **4.75×** |

## What actually mattered

The first-draft DC-only decoder was slower than jSquash on images > 200×200
because the Huffman decoder linear-scanned through codes for every symbol.
Replacing that with the **textbook 256-entry fast lookup table** flipped the
result entirely:

| Fixture              | DC-only (linear scan) | DC-only (fast lookup) | Δ         |
| -------------------- | --------------------- | --------------------- | --------- |
| 64×64 4:4:4          | 0.031 ms              | 0.008 ms              | 3.9× faster |
| 128×128 4:2:0        | 0.078 ms              | 0.014 ms              | 5.6× faster |
| 640×480 4:4:4        | 5.59 ms               | 0.80 ms               | 7.0× faster |
| 640×480 4:2:0        | 3.68 ms               | 0.47 ms               | 7.8× faster |
| 1280×720 4:2:0       | 8.82 ms               | 1.15 ms               | 7.7× faster |

The lesson: our `huffman_decode` was called once per block × 64 AC positions
× 3 components × thousands of MCUs. A 10× speedup there dominates everything
else.

## Remaining headroom

1. **Buffered BitReader.** Current `peek_bits` does save/restore over
   `read_bit` to stay correct — the naïve-buffer version I tried breaks on
   corner cases. With a proper rolling u32 window we could get another
   1.5–2× on the hot path. Left for a follow-up session.
2. **ispeek → consume without re-read.** Today `consume_bits(8)` calls
   `read_bit` 8 times. A buffered reader lets us shift the cached register
   and skip 7 branches per symbol.
3. **Precompute AC-skip table.** A second 256-entry table sized by the
   (R, S) code so we can also skip AC pairs without per-bit loops.
4. **Bigger fixtures + photo content.** Gradient-and-shapes PIL output
   compresses more aggressively than real photos; measure against 12MP
   camera JPEGs for a worst-case number.
