# Bench results — 2026-04-24 (local Node baseline)

Purpose: confirm shipped code still hits the perf targets in `CLAUDE.md`
and identify where the next useful optimization lever is. Re-run after
the doc-sync that moved DC-only + Rust WASM from "deferred" to "shipped".

## Setup

- Host: Apple M2 Pro, Darwin 24.6.0 arm64, Node v24.4.1, pnpm 9.15.0.
- Runner: `pnpm bench` (vitest 2.1.9 `bench` mode).
- Fixtures: `test/fixtures/*.jpg` (10 JPEGs: 64×64 → 1280×720, mix of
  4:2:0, 4:4:4, progressive, CMYK, gray, noise).
- WASM: `packages/core/wasm/paleta_core_bg.wasm`, loaded via
  `initWasm(await readFile(...))` at module top.

## EXIF-thumbnail fast path

Added `bench/exif-thumb.bench.ts` and generated a fixture
(`test/fixtures/paleta-scene-with-exif-thumb.jpg`, 34 KB) that embeds a
real 160×120 JPEG thumbnail in the APP1/Exif segment of the 1280×720
scene. The APP1 parser walks once per call; the embedded thumbnail is a
~2 KB JPEG that jSquash decodes at trivial cost.

| Benchmark                                                   | hz         | mean       |
| ----------------------------------------------------------- | ---------- | ---------- |
| `extractExifThumbnail(bytes)` — parse only                  | 1,709,438  | 0.0006 ms  |
| e2e `getPalette` — EXIF-thumb fast path                     | 2,545      | 0.39 ms    |
| e2e `getPalette` — DC-only (same source image)              | 953        | 1.05 ms    |
| e2e `getPalette` — full-decode (same source image)          | 148        | 6.75 ms    |

**EXIF-thumb is 2.67× over DC-only and 17.16× over full-decode** on this
fixture. The parse itself is free (<1 μs); the win comes from decoding a
2 KB / 160×120 image instead of a 32 KB / 1280×720 one. The caveat is
real-world hit rate: camera-origin and CDN-sanitized JPEGs typically
carry APP1 thumbnails; freshly-encoded web assets from Sharp/PIL often
strip them. Measure in-the-wild coverage before relying on this as the
default.

## How this lines up with the perf targets (CLAUDE.md)

Targets were written against jSquash-only + pure-JS Wu. DC-only and
Rust/WASM have since shipped and collapsed the numbers by ~10×.
Node-measured means below, targets are Workers warm means — these
aren't the same runtime, but Node-vs-workerd delta is ≤30 % (see
`2026-04-20-workers-isolate.md`). So Node numbers are a credible
lower bound.

| Path                 | Target (warm) | Node measured | Ratio       |
| -------------------- | ------------- | ------------- | ----------- |
| DC-only JPEG 1080p   | 10–25 ms      | **1.06 ms**   | ~10× under  |
| Full decode 1080p    | 40–80 ms      | **6.61 ms**   | ~6–12× under|
| EXIF-thumb JPEG 1080p| 5–15 ms       | **0.39 ms**   | ~15× under  |

**Targets are stale.** They predate the current kernel by enough that
they're no longer useful as a ceiling. Either tighten them for v1.0
(e.g. "DC-only 1080p warm ≤ 3 ms") or remove them and publish a
table of *actual* numbers as the commitment.

## Where's the next optimization lever?

Break the 1280×720 DC-only e2e down:

| Stage                          | Approx share of 1.06 ms |
| ------------------------------ | ----------------------- |
| DC-only decode                 | 0.77 ms  (73 %)         |
| Everything else (hist + Wu + OKLab + sort) | 0.29 ms  (27 %)    |

Decode dominates. A further 2–3× speedup in Wu saves ~0.1 ms; a 2×
speedup in DC-only decode saves ~0.4 ms. Conclusion:

1. **Rust `panic=unwind` port is not urgent.** The Wu hot path is
   already ~3× over JS, and the bench sits ~0.25 ms/call — another
   2× costs toolchain complexity to save ~0.1 ms. Revisit when
   someone actually asks for the reliability mode, per ADR-003.
2. **EXIF-thumb path is now benchmarked.** 0.39 ms e2e — 2.67× over
   DC-only. When an APP1 thumbnail is present, this is the obvious
   default. Remaining open question is real-world hit rate on the
   JPEGs users actually point at us.
3. **Cache hit path isn't benchmarked.** Target is 1–3 ms; at the
   current DC-only 1.06 ms, cache might not pay off on warm single-box
   workloads. Worth measuring in workerd with `caches.default`.

## Caveats on these numbers

- Single-box, no cold start, no WASM parse cost in the call path
  (initialized at module top).
- M2 Pro is ~1.5–2× faster than Cloudflare's hypervised cores. Expect
  workerd numbers to be higher; see the 2026-04-20 isolate run for the
  translation factor (Node → workerd is within ~15–30 %).
- `vitest bench` measures `hz` with high-precision `performance.now()`,
  unlike workerd which rounds to 1 ms without the paid compat flag.
- Noise fixtures (1024×1024) are synthetic; real photos produce similar
  histograms but slightly different Wu variance split costs.
